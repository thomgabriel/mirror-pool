# SOAK Implementation Plan (F2a)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** `crates/soak` — a binary that drives one complete live protocol exercise (withdraw
round k=17 + native-stake round k=10) against a local `solana-test-validator` over real RPC,
asserts A1–A7 from chain reads, and emits `docs/soak-report.md`; plus the hand-framed
`docs/SOAK.md` embedding a real captured run.

**Architecture:** per the fork-approved spec `docs/superpowers/specs/2026-07-20-soak-design.md`
(read §2–§5 once — the run shape, assertion set, and honesty ledger bind every task). The
LiteSVM e2e (`crates/sdk/tests/e2e.rs`) is the sequence blueprint; every send becomes an RPC
send-and-confirm. No LiteSVM anywhere in this crate.

**Tech Stack:** Rust 2021; `solana-client` (blocking `RpcClient`) + `solana-sdk` (versioned tx,
ALT instructions); existing `sdk`, `prover`, `effective-k`, `pool-program` (types only) crates.

## Global Constraints

- **Assertion honesty is the review bar:** every claim in the report and SOAK.md maps 1:1 to a
  named assertion (A1–A7) computed from RPC reads (`get_transaction` with
  `max_supported_transaction_version: 0`, `get_account`, balances). Nothing is asserted from
  client-side bookkeeping alone.
- **Fail closed:** any failed assertion or phase → non-zero exit; the report is written with the
  failure marked and a `RUN FAILED` banner — never a partial report that reads as a pass.
- **Blockhash discipline:** fetch a recent blockhash immediately before each send (helper
  enforces it); never before the proof phase.
- **No new on-chain code; no SDK signature changes.** Additions to `crates/soak` only.
- **The effective-k number printed in report/doc is the crate's output verbatim** — never a
  hand-written formula (spec C1).
- Gate per task: `cargo build -p soak && cargo fmt --check && cargo clippy --all-targets -- -D
  warnings` + `cargo deny check` (Task 1 adds solana-client's tree — resolve/justify any new
  advisory in deny.toml with the honest-justification style the file already uses) + the task's
  live smoke run against a validator. `cargo test --workspace` must stay green (soak adds no
  tests; it must not break the build).
- Live-validator workflow for smoke runs: `anchor build` first, then in a background shell
  `solana-test-validator --reset --bpf-program 7oHnDkpPbhPacDfqzF38caM3eo1Xo7cBmFugNXJurnn3
  target/deploy/pool_program.so` (give it ~10 s to boot; kill it when the smoke run ends). Run
  the soak binary FOREGROUND with a generous timeout (600000ms).
- Conventional commits, one per task.

### Task 1: Crate scaffold + RPC plumbing + preflight + setup phase + report skeleton

**Files:**
- Create: `crates/soak/Cargo.toml`, `crates/soak/src/main.rs`, `crates/soak/src/rpc.rs`,
  `crates/soak/src/report.rs`, `crates/soak/src/phases/mod.rs`,
  `crates/soak/src/phases/preflight.rs`, `crates/soak/src/phases/setup.rs`
- Modify: root `Cargo.toml` only if the workspace `members` list is not a glob (check; if
  `members = ["crates/*", ...]` no change needed), `deny.toml` only if `cargo deny check`
  surfaces new advisories (with justification comments).

**Interfaces (Tasks 2–3 rely on these exact items):**
- `rpc::Ctx { client: RpcClient, operator: Keypair, report: Report }` — constructed in main.
- `rpc::send_ixs(ctx: &Ctx, label: &str, ixs: &[Instruction], signers: &[&Keypair]) ->
  Result<Signature>` — legacy message; fetches a fresh blockhash internally; send_and_confirm;
  records `(label, signature)` into the report's tx table.
- `rpc::send_v0(ctx: &Ctx, label: &str, ixs: &[Instruction], alt: &AddressLookupTableAccount,
  signers: &[&Keypair]) -> Result<Signature>` — compiles `v0::Message::try_compile` with the ALT,
  wraps in `VersionedTransaction`, fresh blockhash, send_and_confirm, records into the report.
- `rpc::create_and_fill_alt(ctx: &Ctx, addresses: &[Pubkey]) ->
  Result<AddressLookupTableAccount>` — create ALT, extend in chunks of ≤ 20 addresses (safe
  margin under the ~30/tx wire ceiling), wait until ≥1 slot past the LAST extend's landed slot
  (poll `get_slot`), then `get_account` the table and deserialize to return an
  `AddressLookupTableAccount { key, addresses }` for message compilation.
- `report::Report` — accumulates sections; `Report::assertion(id: &str, desc: &str, pass: bool,
  evidence: String)`; `Report::finish(path)` writes `docs/soak-report.md` (header with date,
  git commit, program id, validator version from `get_version`; phase timings; the tx table;
  the assertion table; `RUN PASSED` only if every assertion passed, else `RUN FAILED`).
- `phases::preflight::run(ctx) -> Result<()>` and `phases::setup::run(ctx) -> Result<SetupOut>`
  where `SetupOut { vote_account: Pubkey, withdraw_pool: Pubkey, withdraw_vault: Pubkey,
  stake_pool: Pubkey, stake_vault: Pubkey, mints: (Pubkey, Pubkey), stake_denomination: u64 }`.

- [ ] **Step 2: rpc.rs — the three helpers + Ctx**

Implement per the Interfaces block. Key details:
- Fresh blockhash: `let bh = ctx.client.get_latest_blockhash()?;` inside each send helper —
  never cached across calls (spec's binding blockhash discipline).
- `send_v0`: `v0::Message::try_compile(&payer, ixs, &[alt.clone()], bh)` →
  `VersionedTransaction::try_new(VersionedMessage::V0(msg), signers)` →
  `send_and_confirm_transaction`.
- `create_and_fill_alt`: `create_lookup_table(authority, payer, recent_slot)` (use
  `get_slot()?` for recent_slot; on "not a recent slot" errors retry once with a fresh slot),
  then `extend_lookup_table` per chunk via `send_ixs`, capturing each landed slot from
  `get_signature_statuses`/`get_transaction` (slot field); after the last extend, poll
  `get_slot()` until `> last_extend_slot`, then fetch+deserialize the table
  (`solana_sdk::address_lookup_table::state::AddressLookupTable::deserialize`) and build the
  return value from its addresses.
- All errors bubble as `anyhow`-style (or a small `SoakError`) with the phase+label in the
  message — actionable, never `unwrap` on RPC results.

- [ ] **Step 3: report.rs + main.rs skeleton**

`main.rs`: parse `--url` (default `http://127.0.0.1:8899`) and `--report`
(default `docs/soak-report.md`); build `Ctx`; run phases in order (preflight, setup — Tasks 2–3
append the rounds); on any error: mark the report FAILED, write it, exit(1). `report.rs` per the
Interfaces block — stable markdown section order so runs diff cleanly.

- [ ] **Step 4: preflight.rs**

Checks, each with an actionable failure message (spec §2.1): RPC reachable (`get_version` —
also recorded in the report header); program account exists + `executable` at
`pool_program::ID`; the four artifact paths exist (`circuits/build/membership_js/membership.wasm`,
`membership.r1cs`, `membership.zkey`, `verification_key.json` — message points at
`bash circuits/scripts/setup.sh`); operator funding — loop `request_airdrop(2 SOL)` +
confirm until balance ≥ 13 SOL (faucet caps per call).

- [ ] **Step 5: setup.rs**

- Vote account: RPC port of the fixtures' `create_validator_vote_account`
  (`crates/sdk/tests/e2e.rs` ~line 114): fund a node keypair, then
  `vote::instruction::create_account_with_config` with `VoteInit { node_pubkey, authorized_voter:
  node, authorized_withdrawer: node, commission: 0 }`, rent for
  `VoteStateVersions::vote_state_size_of(true)`, 3 signers via `send_ixs`.
- Two pools via `sdk::build_initialize_pool_ix` — NOTE its real signature takes 10 args with the
  pool/vault/round PDAs and mint FIRST; transcribe the call shape (and the PDA derivations) from
  `crates/sdk/tests/e2e.rs`'s setup lines, do not guess: withdraw (unique mint pubkey, DENOMINATION =
  2_000_000, k_floor = 2, kind 0, validator = default, FEE = 1_000) and stake (kind 1, the vote
  account, STAKE_FEE = 5_000, denomination = `MIN_STAKE_DELEGATION + rent_for(STAKE_ACCOUNT_SIZE)
  + STAKE_FEE + 1_000_000` — compute rent via `client.get_minimum_balance_for_rent_exemption(
  pool_program::invariants::STAKE_ACCOUNT_SIZE)`, mirroring `stake_pool_denomination` in the
  test fixtures but with LIVE rent, not `Rent::default()`).
- Return `SetupOut`.

- [ ] **Step 6: gate + smoke + commit**

Gate: `cargo build -p soak && cargo fmt --check && cargo clippy --all-targets -- -D warnings &&
cargo deny check` (fold any new advisory into deny.toml with justification). Smoke: launch the

### Task 2: Withdraw round (k = 17) + assertions A1–A7

**Files:**
- Create: `crates/soak/src/phases/withdraw_round.rs`, `crates/soak/src/assertions.rs`
- Modify: `crates/soak/src/main.rs` (wire the phase; remove Task 1's early exit),
  `crates/soak/src/phases/mod.rs`

**Interfaces:**
- Consumes Task 1's `rpc::*`, `report::*`, `SetupOut`.
- Produces `assertions::{assert_signer_set, assert_vault_delta, assert_uniform_payouts,
  assert_nullifiers_spent_and_probe, assert_round_lifecycle, effective_k_section,
  assert_envelope}` — Task 3 reuses every one for the stake round (write them round-kind-generic
  where the spec's definitions differ only by payout target).

- [ ] **Step 1: withdraw_round.rs — the flow (blueprint: e2e.rs's withdraw path, RPC-ified)**

1. 17 `Note::new()` + client `MerkleTree`; 17 `build_deposit_ix` sends (fresh blockhash each via
   `send_ixs`); root final after the last deposit.
2. **One prove-and-commit loop (fork plan-gate fix — the SDK builder proves INTERNALLY; there is
   no pre-generated-proof parameter):** per intent, with fresh `Pubkey::new_unique()` recipient +
   relayer recorded, call `sdk::build_commit_intent_ix(..., FEE, round_id,
   MembershipArtifacts { wasm_path, r1cs_path, zkey_path })` — the builder computes the ext-data
   hash from `(recipient, relayer, fee)` and runs `prove_membership` inside (THIS call is the
   pure-Rust proving being exercised) and returns `CommitIntentBuild { instruction,
   public_inputs }` — then send `.instruction` via `send_ixs`. Time the whole loop as the proof
   phase (sequential acceptable). Do NOT call `prover::prove_membership` separately — that would
   double the ~4–5 min proving or dead-end hunting for a proof parameter. Keep intent #0's
   `CommitIntentBuild` around for the probe.
3. Record every signature.
4. **A4 negative probe NOW (round still Open, current round_id):** re-send intent #0's RETAINED
   `CommitIntentBuild.instruction` verbatim (no re-proving needed); assert the send FAILS and
   that intent #0's PDA account data is byte-identical before/after (fetch before + after and
   compare) — per spec, no error-code pinning.
5. ALT: `create_and_fill_alt` with the 5 infra keys it needs beyond static compilation +
   all 51 per-intent keys (pass the full triple list; let `try_compile` decide what resolves via
   ALT — the table just has to contain the per-intent keys; including pool/round/vault is
   harmless and exercises bigger tables).
6. Execute: `[ComputeBudgetInstruction::set_compute_unit_limit(1_000_000),
   build_execute_round_ix(pool, vault, operator.pubkey(), 0, &sorted_or_recorded_triples)]` via
   `send_v0`. (Intent order: any order — ordering is NOT constrained on-chain; pass them in
   commit order.)
7. Run assertions A1–A7 (Step 2) with the vault balance snapshotted PRE-execute.

- [ ] **Step 2: assertions.rs — each maps to the spec's A-list, all from RPC reads**

- `assert_signer_set(ctx, sig, forbidden: &[Pubkey], expected_signers: &[Pubkey])` (A1): fetch
  `get_transaction_with_config(sig, {encoding: Base64, max_supported_transaction_version: 0})`;
  decode the versioned message; signers = first `header.num_required_signatures` static account
  keys. Assert == `expected_signers` (the cranker/operator only) and that NO forbidden key
  (every recipient, relayer, and the operator's depositor role is the operator itself — list
  recipients+relayers) appears in the signer set. Evidence string: the signer list verbatim.
- `assert_vault_delta(ctx, vault, pre: u64, k, denomination)` (A2): post = `get_balance(vault)`;
  assert `pre - post == k * denomination`.
- `assert_uniform_payouts(ctx, pairs: &[(Pubkey, u64)])` (A3): for withdraw, recipients each
  exactly `denomination - fee` and relayers each exactly `fee` (fresh keys start at 0, so
  absolute balance == credit); build the pairs list accordingly. Evidence: the k identical
  amounts.
- `assert_nullifiers_spent_and_probe` (A4): all k nullifier PDAs exist (account present, owner =
  pool_program) — the probe result from Step 1.4 is folded into this assertion's evidence.
- `assert_round_lifecycle(ctx, pool, round_id)` (A5): round PDA at `round_id` deserializes with
  `state == Executed` (borsh via `pool_program::round::Round::try_deserialize` on the account
  data); round `round_id+1` exists, `Open`, `intent_count == 0`.
- `effective_k_section(report, k)` (A6): `RoundComposition::new(vec![operator_funder_id; k])` →
  `anonymity_report(...)` → print the struct's fields verbatim into the report with the spec's
  fixed honesty sentence (solo run ⇒ collapse to 1.0 is expected and disclosed).
- `assert_envelope(ctx, sig)` (A7): from the same fetched tx: resolved key count =
  static keys + `meta.loaded_addresses.writable.len() + .readonly.len()` ≤ 64;
  `meta.compute_units_consumed` recorded. Evidence: both numbers.
